# arXiv-Paper-Assistant 用户体验反馈与问题总结

本文档汇总实际使用中的优点与待改进点，便于向开发者反馈或作为后续迭代参考。

---

## 一、优点

### 1. 交互体验感不错：智能委派与协同规划

- 具备「团队主管 + 外网情报员 + 本地 RAG 专家」的**智能委派与协同规划**机制。
- 能根据用户意图自动拆分任务、委派给合适的 Agent，而不是单纯一问一答。
- 例如：用户说「帮我下载/阅读几篇扩散模型核心论文」时，系统会自动执行「检索 → 挑选论文 → 批量下载 → 尝试索引」的流程。
- 这种「协同规划」的体验类似与一个小团队合作，**明显优于传统单 Agent 的纯问答助手**，对不熟悉 arXiv 或不想手动查 ID 的用户非常友好。

### 2. 交互界面设计也不错：有颜色框

- CLI 中对话消息、工具调用、团队消息等均有**带颜色的框和分区**，信息层次清晰。
- 能一眼区分「谁在说话、在做什么」，阅读压力小、可读性强，适合长对话和复杂任务的跟踪。

## 二、问题与改进建议

### 1. 论文引用 / arXiv ID 与真实论文不一致（严重）

- **现象**：模型在对话中给出如「Rectified Flow (2305.11367)」的引用，但实际下载的 `2305.11367.pdf` 标题并非 Rectified Flow，而是另一篇论文。
- **本质**：LLM 幻觉 + 缺少「标题 vs ID」一致性校验；下载逻辑按给定 ID 访问 `https://arxiv.org/pdf/<ID>.pdf` 本身无误，但上游提供的 ID 错误导致用户误以为「系统下错论文」。
- **建议**：在检索/推荐或下载前后增加校验（如访问 `https://arxiv.org/abs/<ID>` 核对标题是否包含模型声称的论文名），不一致时提示用户确认，避免静默下载错误论文。

---

### 2. 仅用语义块向量检索的 RAG，对数学论文不够理想（方向问题）

- **现状**：PDF → 语义分块（约 800 字）→ `bge-m3` 向量 → LanceDB 检索，在一般技术论文场景是可用的主流做法。
- **局限**：对数学论文而言，缺少「定理 / 证明 / 公式 / 章节」等**结构化元数据**，难以支撑高质量的结构化推理与精读。
- **建议**：中长期可考虑「**结构化索引（定理、证明、符号关系等）+ 语义向量检索**」结合，先按结构筛再按语义精排，命中后再回看更大范围原文。

---

### 3. 嵌入服务（Ollama + bge-m3）频繁 Request timed out（工程健壮性）

- **现象**：本机 `ollama serve` 与 `ollama run bge-m3 "hello"` 均正常，但对大论文做语义分块 + 批量嵌入时多次出现 `WARNING Request timed out.`，导致下载成功但索引未完成，检索时仍为 `Found 0 documents`。
- **原因**：大论文分块数量多，单次或总体嵌入耗时超过客户端/库的默认超时；或资源紧张导致排队超时。
- **建议**：调大嵌入相关 HTTP 超时、增加重试、优化分块策略（如适当增大 `chunk_size` 减少请求数），并在超时或失败时给出明确提示（如「嵌入服务超时，请确认 Ollama 已运行且资源充足」）。

---

### 4. 环境与依赖配置不够一键化（次要）

- **现象**：按 README 安装后仍缺包（如 `sqlalchemy`、`chonkie[semantic]`、`flatbuffers`、`ollama`、`openai` 等），需多次手动补进 `requirements.txt` 才能跑通；路径曾硬编码为 Windows `C:/Users/username/tmp/...`，在 macOS 上需改为基于 `Path(__file__).parent` 的相对路径。
- **建议**：在官方 `requirements.txt` 中补全所有传递依赖；路径统一使用项目内相对路径，便于跨平台与不同用户环境。

---

## 三、简要汇总

| 类型     | 内容                                                                 |
|----------|----------------------------------------------------------------------|
| **优点** | 智能委派与协同规划带来良好交互体验；终端 CLI 分区与颜色框可读性强。 |
| **严重** | 模型给出的论文引用/arXiv ID 与真实论文不一致，需增加校验与提示。   |
| **方向** | 数学论文场景建议引入结构化元数据与定理级 RAG。                      |
| **工程** | 嵌入服务超时与索引大论文体验需通过超时、重试、分块策略改进。         |
| **配置** | 依赖与路径建议一键化、跨平台友好。                                  |
